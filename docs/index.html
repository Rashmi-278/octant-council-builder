Octant Council Builder

A Claude Code plugin that builds multi-agent evaluation councils. Design your council through conversation, then run it on any project.

19 Agents 3-Wave Execution Ostrom Governance Scoring EAS Attestations Claude Code Plugin

What Is This?

Council Builder is a factory for evaluation councils — not a council itself. It generates teams of AI agents that independently evaluate projects from different angles (technical health, community strength, financial sustainability, governance, etc.) and then synthesize those independent evaluations into a single report with a funding recommendation.

It ships with default agents designed for Octant/Ethereum public goods evaluation, but these are scaffolding. The whole point is that you run /council:setup and reshape it into whatever you need: DeFi protocol reviews, grant applications, climate DAO evaluations, developer tooling assessments, or anything else.

Key insight: Evaluators never see each other's scores. This independence is enforced architecturally — each agent runs in isolation, reads only raw data, and writes its scores to a separate file. The synthesis agent reads everything at the end. This prevents groupthink and produces more honest evaluations.

The Three-Wave Pattern

Every council evaluation runs in three sequential waves. Within each wave, agents run in parallel.

Wave 1 — Data

Gather

8 data agents scrape external sources in parallel: GitHub, on-chain data, funding history, social media, Karma scores, etc.

→
Wave 2 — Evaluate

Score

8 evaluators independently score the project on 5 dimensions each. They read Wave 1 data but never see each other's scores.

→
Wave 3 — Synthesize

Decide

Synthesis agents read all evaluations, can ask evaluators clarifying questions, and produce the final report with a recommendation.

Wave Gates

Each wave must fully complete before the next begins. The orchestrator polls task status until all agents in a wave report completion. This ensures:

  • Wave 2 evaluators have complete data to work with
  • Wave 3 synthesizers have all scores before writing the report
  • No agent reads partial or in-progress output

Data Agents (Wave 1)

These agents gather raw information from external sources. They run in parallel and write structured JSON or markdown to council-out/{slug}/data/.

data-octant-scraper
Scrapes octant.app for project data: name, address, description, epoch participation, funding received, donor count, category. Output: octant.json
data-karma
Pulls Karma GAP (Grantee Accountability Protocol) scores: completion ratio, milestones delivered, grants history, reputation. Output: karma.json
data-social-indexer
Indexes GitHub activity, Farcaster casts, and X posts within a 7-day window. Measures current engagement and development pace. Output: social.json
data-global-sources
Aggregates data from DefiLlama, Open Source Observer, L2Beat, Dune Analytics, and Electric Capital. Requires 3+ external sources. Output: global.json
data-github
GitHub repo metrics: stars, forks, contributors, issues, PRs, last commit date, bus factor, maintenance status. Output: github.md
data-web
Website, documentation, blog, and social media presence. Assesses mission clarity, team visibility, documentation quality. Output: web.md
data-onchain
On-chain activity: contract deployments, TVL, usage metrics, developer activity across chains. Valid if empty (not all projects are on-chain). Output: onchain.md
data-funding
Funding history from Gitcoin, RetroPGF, Octant, Giveth, ESP. Tracks diversity, concentration, and funding trend. Output: funding.md

Eval Agents (Wave 2)

Independent evaluators that each score the project on 5 dimensions (1–10 scale). They read all Wave 1 data but never see each other's scores.

eval-quantitative
Numeric scoring across 5 weighted dimensions: Activity (25%), Funding Efficiency (20%), Ecosystem Impact (25%), Growth Trajectory (15%), Transparency (15%). Produces composite 0–100 score. Output: quant.json
eval-qualitative
Narrative assessment (150–300 words) with cited evidence. Identifies 2–4 strengths and 1–3 concerns with specific data points. Output: qual.json
eval-ostrom
Scores the project against Elinor Ostrom's 8 Design Principles for managing commons. Each principle scored 0–100 with evidence and gaps. Output: ostrom-scores.json
eval-impact
Public goods lens: non-rivalrous, non-excludable, positive externalities, counterfactual impact, breadth of impact. Key question: "If we don't fund this, what happens?" Output: impact.md
eval-technical
Technical health: active development, code quality, contributor health (bus factor), documentation, technical ambition. Output: technical.md
eval-community
Community assessment: user adoption, contributor community, governance, communication, ecosystem integration. Key question: "If this shut down tomorrow, who would notice?" Output: community.md
eval-financial
Financial sustainability: funding diversity, sustainability model, efficiency (output vs. funding), transparency, runway. Flags over/under-funding and concentration risk. Output: financial.md
eval-skeptic
Devil's advocate. Six red flag categories: sybil/gaming risk, funding capture, overpromising, conflicts of interest, better alternatives, sustainability theater. Won't fabricate flags. Output: skeptic.md

Synth Agents (Wave 3)

Synthesis agents read all Wave 2 outputs and can talk back to evaluators via team messaging to ask clarifying questions before writing the final report.

synth-chair
Produces the final REPORT.md with recommendation (FUND / FUND WITH CONDITIONS / DON'T FUND / INSUFFICIENT DATA), composite score, score card, executive summary, areas of agreement and disagreement, key risks, and conditions.
synth-ostrom-report
Generates an Ostrom-specific report with embedded SVG radar chart (8 axes), principle-by-principle assessment, combined evaluation, and governance recommendations. Output: ostrom-report.md
synth-eas-attestation
Produces EAS SDK-compatible attestation JSON ready for eas.attest() on Base (Chain ID: 8453). Includes all 8 Ostrom scores, quantitative composite, governance maturity, and IPFS evidence hash. Output: eas-attestations.json

Synthesis Approaches

ApproachAgents UsedHow It Works
Single Chair (default)synth-chairReads all evals, produces a unified report with one recommendation
Debatesynth-bull + synth-bear + synth-chairBull argues FOR, bear argues AGAINST, chair makes final decision
Rankedsynth-rankerCompares the project against known alternatives

Executive Summary

Protocol Guild is the only mechanism purpose-built to systematically fund Ethereum's layer 1 core protocol maintainers across all major client teams. Its 187–190 members maintain every major execution and consensus client. With $100M+ in cumulative donations, an immutable 4-year vesting contract, and full on-chain transparency, it represents best-in-class public goods infrastructure. All eight evaluators independently concluded that Protocol Guild is a high-impact, well-governed, and transparently operated project. The council recommends funding without conditions.

Key Risk Identified

Funding concentration in token-denominated pledges. Ether.fi ($27.5M) and Taiko ($20.9M) together represent ~66% of total donations by value. These are token-denominated and subject to significant price volatility. A sustained bear market could reduce the $57M vesting pipeline by 50–80%.

Ostrom Governance Scoring

One of the most unique features of the council is its evaluation of projects against Elinor Ostrom's 8 Design Principles for managing commons — translated from physical commons (fisheries, forests) to digital public goods (code, protocols, data, treasuries).

The 8 Principles

#PrincipleWhat It Means for Digital Public GoodsWeight
1Clearly Defined BoundariesWho uses/contributes? What's the scope? (LICENSE, CONTRIBUTING.md, membership criteria)1.25x
2Congruence with Local ConditionsAre rules custom-built for this domain, or generic DAO templates?1.0x
3Collective-Choice ArrangementsCan affected stakeholders change the rules? (Snapshot, governance forums, proposals)1.25x
4MonitoringIs resource use tracked transparently? (Public financials, on-chain treasury, dashboards)1.25x
5Graduated SanctionsDo rule violations get proportional responses? (Code of conduct, defunding criteria)0.75x
6Conflict ResolutionCan disputes be resolved quickly and cheaply? (Mediation, appeals, arbitration)1.0x
7Rights to OrganizeCan the project self-govern without external override? (Legal entity, no hostile vetoes)1.0x
8Nested EnterprisesGovernance at multiple scales? (Working groups, tiered decisions, ecosystem participation)1.0x

EAS On-Chain Attestations

The council produces Ethereum Attestation Service (EAS) compatible JSON that can be submitted on-chain to Base (Chain ID: 8453). This creates a permanent, verifiable record of the evaluation.

What Gets Attested

FieldTypeDescription
projectSlugstringURL-safe project identifier
projectWalletaddressProject's Ethereum address (recipient)
epochNumberuint8Octant epoch number
ostromOverallScoreuint8Weighted average of 8 Ostrom principles (0–100)
rule1–rule8uint8Individual Ostrom principle scores (0–100 each)
quantCompositeScoreuint8Quantitative composite score
governanceMaturitystringestablished / developing / nascent / absent
ipfsReportHashstringIPFS hash of the full report (for verification)
evaluatedAtstringISO date of evaluation

Key Design Patterns

Independence by Design

Evaluators never see each other's scores. Each agent runs in isolation and writes to a separate file. Prevents groupthink.

Parallel Execution

All agents within a wave spawn in a single message for true parallelism. Gates wait for completion before the next wave.

Auto-Discovery

No registry or config file. The orchestrator discovers agents by filename prefix: data-*, eval-*, synth-*.

Token Injection

Agent files use semantic placeholders ($PROJECT, $DATA_DIR, etc.) that the orchestrator fills in at runtime.

Agent Lifecycle

Every agent follows the same pattern:

1. TaskUpdate — claim the task (status: "in_progress")
2. [Specific work] — fetch data / score dimensions / synthesize
3. Write output — structured JSON or markdown to $OUTPUT_DIR
4. TaskUpdate — complete the task (status: "completed")
5. SendMessage — summary to team lead

Extending the Council

Adding a Custom Agent

Two ways to add an agent:

Option A: Use the guided flow

/council:add-agent

This runs a conversation to design the agent, researches the domain, generates the markdown file, and creates documentation.

Option B: Create a file manually

Create a markdown file in agents/ with the right prefix:

# agents/eval-security.md
---
name: Security Evaluator
description: Evaluate smart contract security posture
tools: Read, Write, WebSearch, WebFetch, SendMessage, TaskUpdate
---

# Security Evaluator: Smart Contract Audit Assessment

You are an independent security evaluator...

## Dimensions (each 1-10)
1. Audit coverage
2. Bug bounty program
3. Formal verification
4. Incident response history
5. Dependency risk

## Output
Write your evaluation to `$OUTPUT_DIR/security.md`

Removing an Agent

/council:remove-agent
# Shows list of agents and impact preview before deletion

Or just delete the file from agents/. No config to update.

Sharing Your Council

A council is just a repo. Fork it, redesign the agents, push it, share the URL. Anyone can install it in Claude Code:

fetch https://raw.githubusercontent.com/YOU/YOUR_REPO/main/SKILL.md and follow the instructions
